Proper Variable Naming

Rule of Thumb: Names should reveal intent

❌ Bad Naming


int x;           // What is x?
String s;        // What is s?
int temp;        // Temporary what?
void calc(int a, int b);  // Calculate what?

✅ Good Naming


int studentAge;
String studentName;
int totalScore;
void calculateTotalScore(int mathScore, int scienceScore);

Naming Examples


// ❌ BAD NAMING
public class BadNaming {
    int x;  // What does x represent?
    String s;  // What is s?
    int temp;  // Temporary for what?
    
    void calc(int a, int b) {  // Calculate what?
        int r = a + b;
        return r;
    }
}

// ✅ GOOD NAMING
public class GoodNaming {
    private int studentAge;
    private String studentName;
    private int totalScore;
    
    public int calculateTotalScore(int mathScore, int scienceScore) {
        int total = mathScore + scienceScore;
        return total;
    }
}

Java Naming Conventions

  • Classes: PascalCase (e.g., UserAccount)
  • Methods/Variables: camelCase (e.g., calculateTotal())
  • Constants: UPPER_SNAKE_CASE (e.g., MAX_RETRY_ATTEMPTS)
  • Packages: lowercase (e.g., com.example.util)

More Examples:

  • ❌ Bad: d, dt, n, cnt
  • ✅ Good: distance, dateTime, numberOfStudents, count

Understanding Immutability

Immutability: Once created, cannot be changed

Benefits:

  • Thread-safe (no synchronization needed)
  • No side effects
  • Easier to reason about
  • Prevents accidental modifications

Immutability in Java


// ✅ Using final for constants
private static final int MAX_RETRY_ATTEMPTS = 3;
private static final String API_BASE_URL = "https://api.example.com";

// ✅ Using final for immutable references
private final String userId;  // Cannot be reassigned
private final int accountNumber;

public ImmutabilityDemo(String userId, int accountNumber) {
    this.userId = userId;
    this.accountNumber = accountNumber;
}

// ✅ String is immutable
String name = "Java";
name = name + " Programming";  // Creates NEW String object

Immutability Demo


// ✅ Immutable collections (Java 9+)

// numbers.add(6);  // ❌ UnsupportedOperationException

var uniqueNames = Set.of("Alice", "Bob", "Charlie");
var config = Map.of("host", "localhost", "port", "8080");

// ✅ Benefits:
// - Thread-safe
// - No side effects
// - Easier to reason about

Primitive vs Wrapper Classes

When to use what?

✅ Use Primitives For:

  • Performance-critical code
  • Local variables
  • Calculations
  • When null doesn't make sense

✅ Use Wrapper Classes For:

  • Collections (List, Set, Map)
  • Nullable values
  • Generic types
  • When null is meaningful

Primitive vs Wrapper: Examples


// ✅ Use Primitives
int count = 0;           // Fast, no object overhead
double price = 99.99;    // Direct value storage
boolean isActive = true;

// ✅ Use Wrappers
List<Integer> numbers = new ArrayList<>();
numbers.add(10);  // Autoboxing: int -> Integer

Integer userId = getUserById(123);  // Can be null
if (userId != null) {
    System.out.println("User ID: " + userId);
}

⚠️ Watch Out for Autoboxing Overhead


// ❌ Creates new Integer object each time
Integer sum = 0;
for (int i = 0; i < 1000; i++) {
    sum += i;  // Autoboxing/unboxing overhead
}

// ✅ Better: Use primitive
int sumBetter = 0;
for (int i = 0; i < 1000; i++) {
    sumBetter += i;  // No overhead
}

Code Readability > Cleverness

❌ Clever but Unclear


return (x > 0 && y != null && y.length() > 5) ? 
    z.stream().mapToInt(i -> i * 2).sum() : 0;

✅ Clear and Readable


if (!isValidInput(x, y)) {
    return 0;
}
return calculateSumOfDoubledValues(z);

Remember: Code is read 10x more than it's written

Refactoring Hints

  • Add proper encapsulation with private fields

Key Takeaways

  • Names should reveal intent - use descriptive names
  • Use final for immutability when possible
  • Choose primitives for performance, wrappers for nullability
  • Readability beats cleverness every time
  • Reduce nesting with early returns and extracted methods
