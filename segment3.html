Cohesion & Coupling

Cohesion: How well do class members belong together?

  • High Cohesion: Class does one thing well ✅
  • Low Cohesion: Class does multiple unrelated things ❌

Coupling: How dependent are classes on each other?

  • Loose Coupling: Classes are independent ✅
  • Tight Coupling: Classes are highly dependent ❌

Cohesion Example

❌ Low Cohesion


public class Utility {
    boolean isValidEmail(String email) { }
    void sendEmail(String to, String subject) { }
    int calculateTax(double amount) { }
    void printReport() { }
}

Multiple unrelated responsibilities

✅ High Cohesion


public class EmailValidator {
    boolean isValidEmail(String email) { }
    boolean isValidDomain(String email) { }
    String extractDomain(String email) { }
}

Coupling Example

❌ Tight Coupling


public class OrderServiceBad {
    private CreditCardProcessor processor = 
        new CreditCardProcessor();  // Hard dependency!
    
    void processOrder(Order order) {
        processor.charge(order.getAmount());
    }
}

Hard to test, hard to change

✅ Loose Coupling


public class OrderService {
    private PaymentProcessor processor;  // Interface
    
    public OrderService(PaymentProcessor p) {
        this.processor = p;  // Dependency injection
    }
    
    void processOrder(Order order) {
        processor.processPayment(order.getAmount());
    }
}

Easy to test, easy to change

Encapsulation: Data Hiding

Principle: Hide internal details, expose only what's necessary

Benefits:

  • Data integrity through validation
  • Flexibility to change implementation
  • Security through controlled access
  • Maintainability with single point of change

❌ Bad: No Encapsulation


public class BadStudent {
    public String name;      // Direct access - anyone can modify
    public int age;          // No validation
    public double gpa;       // Can be set to invalid values
    
    // No control over data integrity
}

// Usage - problems!
BadStudent student = new BadStudent();
student.age = -5;        // Invalid!
student.gpa = 10.0;     // Invalid!

✅ Good: Proper Encapsulation


public class GoodStudent {
    private String name;    // Private - hidden
    private int age;
    private double gpa;
    
    // Constructor with validation
    public GoodStudent(String name, int age, double gpa) {
        setName(name);
        setAge(age);
        setGpa(gpa);
    }
    
    // Getters - controlled access
    public String getName() { return name; }
    public int getAge() { return age; }
    public double getGpa() { return gpa; }
    
    // Setters with validation
    public void setAge(int age) {
        if (age < 0 || age > 150) {
            throw new IllegalArgumentException("Invalid age");
        }
        this.age = age;
    }
}

DRY: Don't Repeat Yourself

Principle: Every piece of knowledge should have a single representation

Ways to Achieve DRY:

  • Abstraction: Extract common behavior
  • Inheritance: Share code through hierarchy
  • Composition: Reuse through objects
  • Methods: Extract repeated logic

DRY Example

❌ Repetitive Code


public void processOrder1() {
    validate();
    calculate();
    save();
    sendEmail("order1@example.com");
}

public void processOrder2() {
    validate();
    calculate();
    save();
    sendEmail("order2@example.com");
}

✅ DRY Code


public void processOrder(String email) {
    validate();
    calculate();
    save();
    sendEmail(email);
}

Writing Meaningful Constructors

✅ Good Constructor Practices:

  • Initialize all required fields
  • Validate input parameters
  • Set sensible defaults
  • Keep constructors simple

❌ Avoid in Constructors:

  • Complex business logic
  • Heavy computations
  • Calling overridable methods
  • Throwing checked exceptions (if possible)

Constructor Example


// ✅ Good Constructor
public class Book {
    private String isbn;
    private String title;
    private String author;
    private boolean isAvailable;
    
    public Book(String isbn, String title, String author) {
        if (isbn == null || isbn.trim().isEmpty()) {
            throw new IllegalArgumentException("ISBN required");
        }
        if (title == null || title.trim().isEmpty()) {
            throw new IllegalArgumentException("Title required");
        }
        
        this.isbn = isbn;
        this.title = title;
        this.author = author;
        this.isAvailable = true;  // Sensible default
    }
}

Book Class


public class Book {
    // Encapsulated fields
    private String isbn;
    private String title;
    private String author;
    private boolean isAvailable;
    
    // Constructor with validation
    public Book(String isbn, String title, String author) {
        this.isbn = isbn;
        this.title = title;
        this.author = author;
        this.isAvailable = true;
    }
    
    // Business logic methods
    public void checkout() {
        if (!isAvailable) {
            throw new IllegalStateException("Book already checked out");
        }
        this.isAvailable = false;
    }
    
    public void returnBook() {
        this.isAvailable = true;
    }
}

Member Class


public class Member {
    private String memberId;
    private String name;
    private List<Book> borrowedBooks;
    private int maxBorrowLimit;
    
    public Member(String memberId, String name, String email) {
        this.memberId = memberId;
        this.name = name;
        this.borrowedBooks = new ArrayList<>();
        this.maxBorrowLimit = 5;
    }
    
    public boolean canBorrowMore() {
        return borrowedBooks.size() < maxBorrowLimit;
    }
    
    public void borrowBook(Book book) {
        if (!canBorrowMore()) {
            throw new IllegalStateException("Reached borrow limit");
        }
        book.checkout();
        borrowedBooks.add(book);
    }
}

Activity Discussion

Where do OOP principles apply?

  • Encapsulation: Private fields with public methods
  • Cohesion: Each class has a single responsibility
  • Coupling: Library depends on Book and Member, but loosely
  • Abstraction: Methods hide implementation details

Key Takeaways

  • High cohesion: Classes should do one thing well
  • Loose coupling: Depend on abstractions, not concrete classes
  • Encapsulation: Hide data, expose behavior
  • DRY: Extract common code into methods/classes
  • Think in objects: Model real-world concepts
